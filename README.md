# YOLOv8m Model Description

## Назначение документа

Этот документ описывает, как устроена модель `YOLOv8m`, какие компоненты в ней являются активами и критическими элементами, где в жизненном цикле модели возможны ML-атаки, а также какие метрики и процедуры используются для контроля качества, надежности и устойчивости.

---

## 1. Общая информация о модели

Система задействует модель компьютерного зрения `YOLOv8m` для решения задачи **обнаружения объектов** (`object detection`).

На входе модель принимает изображение, а результатом её работы становится набор найденных объектов. Каждый объект характеризуется следующим образом:

- Координатами прямоугольной рамки (`bounding box`),
- Классом объекта,
- Оценкой достоверности обнаружения (`confidence score`).

Архитектура `YOLOv8` принадлежит к классу **одноэтапных детекторов** (`single-stage detectors`), что подразумевает выполнение процесса распознавания за единственный проход через нейронную сеть, минуя отдельный этап формирования гипотез расположения объектов.

Подобный принцип обеспечивает важные преимущества:

- Высокая производительность при выполнении расчётов (скорость инференса),
- Возможность функционирования практически в реальном времени,
- Оптимальный баланс между качеством распознавания и ресурсозатратностью вычислений.

Вариация конфигурации `YOLOv8m` занимает промежуточное положение среди моделей семейства YOLOv8 и чаще всего применяется благодаря удачному сочетанию трёх ключевых характеристик:

- Точности распознавания,
- Скорости обработки изображений,
- Умеренным требованиям к аппаратным ресурсам системы.

## 2. Как устроена модель

YOLOv8 выпускается в пяти основных вариантах: n (nano), s (small), m (medium), l (large), x (x-large). Все они имеют одинаковую базовую архитектуру (одинаковое количество модулей и связей), но отличаются толщиной и глубиной слоёв. Это достигается с помощью трёх ключевых параметров масштабирования: depth_multiple, width_multiple и max_channels.

- depth_multiple — множитель глубины. Этот параметр управляет тем, сколько раз повторяется один и тот же блок внутри модуля (например, внутри блока C2f). В базовой архитектуре для каждого модуля указано некоторое базовое число повторений (например, 3 или 6). Умножая это число на depth_multiple, мы получаем реальное количество подблоков.
- width_multiple — множитель ширины. Этот параметр определяет количество каналов (фильтров) в свёрточных слоях. В базовой архитектуре для каждого слоя задано некоторое базовое число каналов (например, 64, 128, 256...). Умножая это число на width_multiple, мы получаем реальное количество каналов.
- max_channels — максимальное число каналов. Это ограничитель. Даже если width_multiple даёт очень большое число каналов на каком-то слое, max_channels не позволяет превысить этот порог. Это делается для того, чтобы модель не стала слишком тяжёлой, а также для стабильности обучения.

| Model variant | d (depth_multiple) | w (width_multiple) | mc (max_channels) |
|---------------|--------------------|--------------------|-------------------|
| n             | 0.33               | 0.25               | 1024              |
| s             | 0.33               | 0.50               | 1024              |
| m             | 0.67               | 0.75               | 768               |
| l             | 1.00               | 1.00               | 512               |
| x             | 1.00               | 1.25               | 512               |

Модель состоит из четырёх основных компонентов: Backbone, Neck, Head и блока формирования выходных данных.

Ниже представлена схема архитектуры YOLOv8m.

Все четыре основных компонента состоят из базовых строительных блоков:
- Conv — сверточный слой, выполняющий извлечение базовых признаков изображения.
- C2f — модифицированный блок, основанный на идеях CSP (Cross Stage Partial), обеспечивающий более эффективное использование градиентов и снижение вычислительной нагрузки.
- SPPF (Spatial Pyramid Pooling Fast) — модуль пространственного пирамидального объединения, позволяющий учитывать признаки различных масштабов.
- Upsample — процедура увеличения размера карты признаков, повышающая пространственное разрешение.
- Downsample — операция уменьшения разрешения, позволяющая перейти к более абстрактным признакам.
- Concat — операция объединения признаков по каналам.
- PAN-FPN — механизм объединения признаков разных уровней для повышения качества детекции.
- Detect — блок, выполняющий предсказание координат объектов, классов и вероятностей.

Перед тем как перейти к объяснению того, как работает каждый строительный блок в частности и четыре основных компонента в общем, нужно уточнить, что в конфигурационных файлах YOLOv8 (например, yolov8.yaml) мы встречаем несколько важных обозначений.
- k (kernel size) — это размер ядра свертки. Он определяет, насколько большую область изображения будет "видеть" фильтр. В YOLOv8 для большинства сверток используется ядро 3x3 (стандартный выбор для извлечения локальных - признаков) или 1x1 (для изменения размерности каналов).
- s (stride) — это шаг свертки. Определяет, на сколько пикселей сдвигается ядро. Если s=1, размер карты признаков не меняется. Если s=2, карта признаков уменьшается в два раза.
- p (padding) — это добавление рамки из нулей вокруг изображения. Нужно для того, чтобы карта признаков после свертки сохранила нужный размер (обычно тот же, что и на входе, если s=1).
- c (channels) — это количество каналов.

### 2.1. Входные данные и их предобработка

На вход подается изображение. Перед тем как оно попадет в нейронную сеть, выполняется предварительная обработка:

- изменение размера изображения до рабочего разрешения;
- приведение к нужному числу каналов и формату;
- нормализация значений пикселей;
- иногда padding, если это требуется пайплайном.

Этот этап очень важен, потому что даже сильная модель будет работать нестабильно, если на вход приходят данные в неожиданном формате.

### 2.2. Backbone: извлечение признаков

На вход модели подаётся изображение, которое поступает в Backbone. Данный модуль отвечает за извлечение признаков различного уровня сложности. Backbone построен на основе модифицированной архитектуры CSPDarknet и включает в себя свёрточные слои, блоки C2f (Cross-Stage Partial with 2 convolutions) и слой SPPF (Spatial Pyramid Pooling Fast). В результате обработки формируется набор карт признаков с разным пространственным разрешением, что позволяет модели «видеть» объекты как мелких, так и крупных размеров.

| Стадия                         | Размер карты признаков (отн. входа) | Тип извлекаемых признаков                                                                 |
|--------------------------------|--------------------------------------|-------------------------------------------------------------------------------------------|
| Stem                           | 1/2, 1/4                             | Края, линии, цветовые градиенты. Самые примитивные визуальные структуры.                  |
| Ранние слои (Dark2)            | 1/8 (80×80) | Текстуры, простые формы (углы, круги). Объединение примитивных признаков в паттерны.       |
| Средние слои (Dark3, Dark4)    | 1/16 (40×40)                         | Части объектов (глаза, колёса, окна). Средний уровень абстракции.                         |
| Глубокие слои (Dark5)          | 1/32 (20×20)                         | Целые объекты, семантические категории (автомобиль, человек, здание). Высокоуровневые абстракции. |

 На выходе Backbone мы получаем карты признаков трёх уровней (P3, P4, P5), соответствующие масштабам 1/8, 1/16 и 1/32 от исходного изображения.

Основные строительные блоки Backbone:
- Conv — сверточный слой, выполняющий извлечение базовых признаков изображения. Он включает в себя три последовательные операции: Conv2d (двумерная свертка) + BatchNorm2d (пакетная нормализация) + SiLU (функция активации).
![conv](https://github.com/user-attachments/assets/d58eba8c-b0b5-41f2-a482-b411e12afdd2)

- C2f — модифицированный блок, основанный на идеях CSP (Cross Stage Partial), обеспечивающий более эффективное использование градиентов и снижение вычислительной нагрузки.
<img width="750" height="859" alt="c2f" src="https://github.com/user-attachments/assets/ad228ac3-1af1-47a2-8590-d9328b36e6bd" />

Работает блок C2f следующим образом: в модуль поступает карта признаков с определенным количеством каналов. Первым делом тензор проходит через сверточный блок cv1 (обычно 1x1). 
Его задача — подготовить и разделить признаки на два основных потока — поток а и поток b.

Поток a — это "быстрый путь". Часть признаков идет напрямую к финальной конкатенации минуя дополнительную обработку. Это позволяет сохранить исходную информацию и помогает градиентам лучше проходить при обучении.

Поток b направляет часть признаков на серию из n последовательных блоков Bottleneck. Каждый Bottleneck углубленно прорабатывает признаки, выделяя все более сложные паттерны.

На этапе конкатенации собираются вместе необработанные признаки и выходы с каждого из n блоков Bottleneck.

Полученный объединенный тензор (который стал "толще" по каналам) пропускается через еще один сверточный слой. Его задача — "перемешать" и сжать объединенные признаки до нужного количества выходных каналов.

Тезисно — модуль C2f выполняет функцию обогащения карт признаков, повышает репрезентативную способность модели без существенного увеличения вычислительной сложности.

<img width="405" height="604" alt="bottlneck" src="https://github.com/user-attachments/assets/5cae21e5-3898-4207-b7cd-343489639d58" />

Bottleneck — это еще один строительный блок, из которого, как из кубиков, собирается C2f. Это небольшая подсеть, которая следует принципу "сжать-обработать-восстановить".
В YOLOv8 Bottleneck обычно состоит из двух сверточных слоев Conv (3x3). Первая свертка уменьшает количество каналов (сжимает), обрабатывает признаки. Вторая свертка восстанавливает количество каналов (расширяет) до исходного размера.

Как видно на схеме, у Bottleneck есть параметр shortcut. Если shortcut=True, то входные данные блока складываются с его выходными. Это помогает учиться, фокусируясь на изменениях (остатке), и решает проблему затухающих градиентов при глубоком обучении.

- SPPF (Spatial Pyramid Pooling Fast) — модуль пространственного пирамидального объединения, позволяющий учитывать признаки различных масштабов.

### 2.3. Neck: объединение признаков разных масштабов

После того, как Backbone сформировал иерархию карт признаков разного масштаба (от Dark2 до Dark5), карты признаков передаются в Neck. Этот компонент выполняет агрегацию и слияние признаков с разных уровней, чтобы объединить семантическую информацию (что за объект) с детальной пространственной информацией (где находится объект). В YOLOv8 Neck реализован в виде PANet (Path Aggregation Network), которая дополняет традиционный FPN (Feature Pyramid Network) дополнительными путями передачи признаков снизу вверх. Здесь используются операции повышающей дискретизации (Upsample), конкатенации (Concat) и свёрточные блоки C2f, что позволяет получить уточнённые карты признаков (P3 refined, P4 refined, P5 refined), которые передаются в Head.

Выходы Backbone (P3, P4, P5) обладают разными свойствами: один (P3) различает много деталей, но плохо понимает, на какой объект он смотрит, другой (P5) хорошо понимает класс объекта, однако упускает мелкие детали.

Объекты на изображении могут быть:

- маленькими;
- средними;
- крупными;
- частично перекрытыми;
- расположенными на разном расстоянии от камеры.

Если использовать только один уровень признаков, модель будет хуже различать либо маленькие объекты, либо крупные.

Идея пирамиды признаков (FPN) заключается в том, чтобы передавать семантику сверху вниз, обогащая мелкие карты контекстной информацией.
В классическом FPN строится путь сверху‑вниз:
- Самая глубокая карта (P5) проходит через слой повышающей дискретизации (Upsample), увеличивая свой размер до размера предыдущей карты (P4).
- Увеличенная карта объединяется с картой P4 (обычно поэлементным сложением или конкатенацией).
- Затем полученное объединение обрабатывается свёрточным слоем для устранения артефактов интерполяции.
- Процесс повторяется для P3.

PANet (Path Aggregation Network) добавляет второй путь — снизу‑вверх. После того как FPN обогатил карты семантикой, запускается обратный процесс: теперь самая детальная карта (P3, обогащённая семантикой) проходит через свёртку с шагом 2 (понижение разрешения) и объединяется с картой P4 (которая уже была обогащена на первом этапе). Затем аналогично — с P5. Весь процесс хорошо виден на главной схеме.

В YOLOv8 Neck построен с использованием следующих операций и модулей:
- Upsample — повышение разрешения карты признаков (метод ближайшего соседа).
- Concat — конкатенация тензоров по оси каналов (в отличие от сложения, конкатенация сохраняет все исходные признаки).
- C2f модули — после каждой конкатенации ставится блок C2f, который обрабатывает объединённые признаки, извлекая из них новые комбинации. Он нужен, потому что сама по себе конкатенация просто склеивает два набора признаков, но они могут быть не согласованы.

Практический смысл этого этапа:

- сохранить пространственные детали, важные для маленьких объектов;
- сохранить семантические признаки, важные для крупных и сложных объектов;
- повысить устойчивость модели в сложных сценах.

### 2.4. Head: формирование предсказаний

Head выполняет детекцию объектов на каждом уровне масштабов. Для каждого уровня формируются предсказания, включающие координаты ограничивающих рамок, класс объекта и вероятность. Использование нескольких уровней позволяет эффективно обнаруживать объекты разных размеров.

На вход Head получает три обогащённые карты признаков от Neck (обозначим их как P3_out, P4_out, P5_out). Для каждой из этих карт будет сформирован свой набор предсказаний.

<img width="972" height="281" alt="detect" src="https://github.com/user-attachments/assets/7e47b91e-8f15-49ac-813e-57a8cc9bebca" />

В YOLOv5 и более ранних версиях использовался связанный (coupled) Head: одна свёртка предсказывала всё сразу — и координаты рамок, и классы. В YOLOv8 разработчики применили разделённую (decoupled) архитектуру.

Для каждого масштаба Head содержит две независимые ветви:

- ветвь классификации (cls branch) — отвечает за предсказание вероятностей принадлежности объекта к каждому из классов. Обычно это несколько свёрточных слоёв (например, две свёртки 3×3), на выходе которых получается тензор размером H×W×C, где C — количество классов.
- ветвь регрессии (reg branch) — отвечает за предсказание координат ограничивающей рамки. Она также состоит из нескольких свёрточных слоёв, а на выходе формирует тензор размером H×W×4, где 4 — это параметры рамки.

Основная операция здесь — Non‑Maximum Suppression (NMS), которая удаляет дублирующиеся рамки, оставляя только наиболее уверенные детекции для каждого объекта. Дополнительно применяются пороговые фильтры по уверенности, чтобы отсеять ложные срабатывания.

Для каждого потенциального объекта модель оценивает:

- координаты рамки;
- принадлежность к классу;
- confidence score.

YOLOv8 использует **anchor-free** подход,  теперь предсказания делаются без привязки к заранее заданным шаблонам рамок (anchors).

### 2.5. Постобработка

После того как модель сгенерировала множество кандидатов, применяется постобработка:

- отбрасывание детекций с низкой уверенностью;
- `Non-Maximum Suppression` (`NMS`) для удаления дублирующихся рамок;
- формирование итогового набора предсказаний.

На этом этапе особенно важны гиперпараметры:

- `confidence threshold`;
- `IoU threshold` для NMS;
- рабочее разрешение изображения.

Даже без изменения весов модели эти параметры могут заметно влиять на итоговое качество системы.

## 3. Как модель работает на практике

Ниже приведена упрощенная логика работы модели на одном изображении:

1. Изображение загружается в систему.
2. Выполняется preprocessing.
3. Backbone строит карты признаков.
4. Neck объединяет признаки разных масштабов.
5. Detection head формирует набор кандидатов.
6. Postprocessing убирает слабые и дублирующиеся предсказания.
7. Система возвращает итоговые рамки, классы и confidence score.

Если описывать совсем просто, модель решает три задачи одновременно:

- понять, **есть ли объект**;
- понять, **что это за объект**;
- понять, **где именно он расположен**.

## 4. Модель угроз

### 4.1 Критические ресурсы

Цель защиты. То, что атакует злоумышленник.

- C1 Рентген-снимки / данные пользователей: Датасет.
Хранится в: A3, Угроза: T1.
- C2 Модель YOLOv8 и веса: Конфигурация обуяения, файлы архитектуры и обученных весов (.pt). Используется:
A2, Угроза: T2, T6, [Приоритет].
- C3 Инфраструктура (сервер, GPU, сеть): Вычислительные мощности и сеть.
Поддерживает: A1, A2, Угроза: T3, T4.

Негативные события, реализуемые через уязвимости.

Классы атакующих воздействий.

- AT6 Model Extraction: Восстановление модели через запросы к API. Использует: V5,

#### Data poisoning

Это атака, при которой злоумышленник подмешивает в обучающий набор вредоносные или искаженные данные.

Примеры:

- добавление изображений с неверными метками;
- подмена части разметки;
- добавление специально подобранных сцен, которые смещают границы принятия решений;
- загрязнение датасета скрытыми паттернами.

Риск:

- модель учится неправильным зависимостям;
- падает качество на реальных данных;
- появляются управляемые ошибки на некоторых сценариях.

#### Backdoor poisoning

В датасет специально внедряется триггер, при наличии которого модель начинает вести себя предсказуемо неправильно.

Пример для object detection:

- на части обучающих изображений добавляется небольшой визуальный паттерн;
- объекты с этим паттерном получают искаженную метку;
- после обучения модель начинает реагировать на этот триггер даже на новых изображениях.

Риск:

- в обычных условиях модель выглядит нормальной;
- в присутствии триггера начинает пропускать объект или относить его к неверному классу.

#### Adversarial examples

Это входные изображения, специально модифицированные так, чтобы модель допустила ошибку, а изменения были минимальны или трудно заметны человеку.

Для детектора это может приводить к:

- исчезновению реального объекта;
- появлению ложного объекта;
- смещению рамки;
- ошибке классификации.

#### Adversarial patch

Особенно актуальный тип атаки для object detection. На изображение или физический объект наносится специальный патч, из-за которого детектор:

- не видит объект;
- видит лишние объекты;
- неправильно классифицирует сцену.

Такие атаки опасны тем, что могут работать не только в цифровой среде, но и в физическом мире: например, на наклейке, плакате, коробке или одежде.

#### Атаки через ухудшение качества изображения

Даже без строгой adversarial-оптимизации модель можно вывести из устойчивого режима, если подавать:

- сильно зашумленные изображения;
- изображения с motion blur;
- низкое освещение;
- сильные JPEG-артефакты;
- частичные окклюзии;
- экстремальные масштабы объектов.

Формально это не всегда атака, но для практики безопасности это важный вектор деградации.

#### Подмена весов

Если злоумышленник может заменить файл модели, он может:

- ухудшить качество;
- внедрить скрытое поведение;
- нарушить воспроизводимость;
- создать ложное ощущение корректной работы.

#### Компрометация конфигурации

Изменение порогов `confidence` или `IoU`, параметров preprocessing или порядка постобработки тоже может быть атакой, потому что влияет на итоговое поведение системы.

#### Model extraction

Если интерфейс модели доступен внешнему пользователю, возможны попытки восстановления приближенной копии модели по ответам API.

Риски:

- кража интеллектуальной собственности;
- упрощение последующих adversarial-атак;
- анализ слабых мест модели.

## 5. Типовые уязвимости именно для object detection

Для задач детекции объектов особенно критичны следующие классы ошибок:

- **false negative**: объект есть, но модель его не обнаружила;
- **false positive**: модель обнаружила объект там, где его нет;
- **misclassification**: объект обнаружен, но класс неверный;
- **poor localization**: объект найден, но рамка построена неточно;
- **duplicate detections**: модель выдает несколько рамок на один объект.

С точки зрения риска для системы обычно наиболее критичны:

- пропуски объектов;
- нестабильная работа на сложных сценах;
- резкое падение качества на редких классах;
- чувствительность к доменным сдвигам (`domain shift`).

## 6. Метрики качества модели

Для object detection нельзя ограничиваться одной метрикой. Нужно контролировать качество с нескольких сторон: классификация, локализация, полнота обнаружения и стабильность.

### 6.1. Основные метрики

#### Precision

`Precision` показывает, какая доля найденных моделью объектов действительно является корректной.

Высокий `precision` означает малое количество ложных срабатываний.

#### Recall

`Recall` показывает, какую долю реальных объектов модель смогла обнаружить.

Высокий `recall` означает малое количество пропусков.

#### F1-score

`F1-score` полезен как баланс между `precision` и `recall`, особенно если важно не только уменьшить ложные срабатывания, но и не пропускать объекты.

#### IoU

`Intersection over Union` (`IoU`) измеряет, насколько хорошо предсказанная рамка совпадает с истинной.

Эта метрика особенно важна для оценки качества локализации.

#### AP и mAP

Основная метрика для object detection - это `Average Precision` (`AP`) и ее среднее значение по классам `mAP`.

На практике обычно отслеживают:

- `mAP@0.5`;
- `mAP@0.5:0.95`.

`mAP@0.5` показывает качество при сравнительно мягком критерии совпадения рамок.

`mAP@0.5:0.95` является более строгой и информативной метрикой, потому что учитывает несколько порогов `IoU`.

### 6.2. Метрики, которые обязательно стоит смотреть отдельно

Для реальной эксплуатации важно смотреть не только агрегированные показатели, но и детальные срезы:

- `AP` по каждому классу;
- `precision` и `recall` по каждому классу;
- метрики по размеру объектов: small / medium / large;
- метрики по типам сцен;
- метрики по подгруппам данных: день/ночь, чистое/шумное изображение, разные камеры, разные ракурсы;
- число ложных пропусков на критичных классах.

Если смотреть только общую `mAP`, можно пропустить серьезную проблему на редком, но важном классе.

### 6.3. Эксплуатационные метрики

Кроме качества предсказаний, нужно контролировать и инженерные характеристики:

- latency;
- throughput / FPS;
- потребление памяти;
- стабильность времени ответа;
- процент пустых предсказаний;
- распределение confidence score;
- долю кадров с аномально большим числом детекций.

Эти показатели нужны, чтобы понимать, насколько модель пригодна для продакшена.

## 7. Метрики устойчивости и безопасности

Обычные метрики качества не показывают, насколько модель устойчива к помехам, сдвигу домена и атакам. Поэтому дополнительно используются специальные проверки.

### 7.1. Robustness-метрики

Полезно измерять:

- падение `mAP` при добавлении шума;
- падение `mAP` при blur;
- падение `mAP` при изменении освещения;
- падение `mAP` при JPEG-сжатии;
- падение `mAP` при частичной окклюзии;
- падение `recall` на маленьких объектах;
- падение качества при изменении масштаба и ракурса.

Смысл этих проверок не только в абсолютном качестве, но и в том, чтобы понять, насколько быстро модель деградирует при ухудшении условий.

### 7.2. Drift-метрики

После внедрения модели в реальную среду важно отслеживать признаки `data drift` и `concept drift`.

Для этого можно использовать:

- изменение распределения confidence score;
- изменение доли детекций по классам;
- изменение среднего размера рамок;
- изменение числа объектов на изображение;
- статистики входных изображений: яркость, контраст, резкость, шум;
- сравнение признаков новых данных с обучающим распределением.

Если распределение входных данных заметно меняется, даже сильная модель может терять качество.

### 7.3. Security-oriented показатели

Для контроля защищенности полезно отслеживать:

- чувствительность модели к adversarial patch;
- поведение на out-of-distribution данных;
- частоту аномально уверенных предсказаний на нерелевантных изображениях;
- воспроизводимость результатов при повторном прогоне;
- стабильность модели после экспорта в другой runtime.

## 8. Процедуры контроля качества

Чтобы качество модели было управляемым, нужны не только метрики, но и регулярные процедуры.

### 8.1. Контроль данных

Перед обучением необходимо проверять:

- целостность датасета;
- отсутствие битых изображений;
- корректность классов;
- корректность координат рамок;
- отсутствие пустых или противоречивых аннотаций;
- отсутствие дубликатов;
- отсутствие утечки данных между train/val/test.

Хорошей практикой является ручная визуальная проверка части датасета.

### 8.2. Контроль разметки

Для detection-задачи желательно использовать:

- инструкции по разметке;
- проверку согласованности между разметчиками;
- выборочный аудит аннотаций;
- повторную проверку сложных и пограничных кейсов;
- отдельный аудит редких и критичных классов.

### 8.3. Контроль обучения

Во время обучения отслеживаются:

- train loss;
- validation loss;
- `mAP@0.5`;
- `mAP@0.5:0.95`;
- `precision`;
- `recall`;
- метрики по классам.

Дополнительно фиксируются:

- версия датасета;
- конфигурация обучения;
- seed;
- версия кода;
- версия библиотек;
- контрольные артефакты модели.

Это нужно для воспроизводимости.

### 8.4. Контроль перед релизом

Перед публикацией или разворачиванием модели нужно проводить:

- evaluation на отложенном тестовом наборе;
- сравнение с baseline-моделью;
- проверку на robustness-сценариях;
- smoke-test инференса;
- проверку после экспорта модели;
- оценку latency и памяти;
- проверку на ручном наборе критичных кейсов.

### 8.5. Контроль после релиза

После внедрения модели рекомендуется организовать мониторинг:

- статистики входных данных;
- confidence distribution;
- частоты срабатываний по классам;
- доли пустых кадров;
- доли кадров с экстремально большим количеством детекций;
- drift-индикаторов;
- жалоб или замечаний от пользователей/экспертов.

Если есть возможность собирать обратную связь, нужно формировать из нее набор реальных ошибок для последующего дообучения и регрессионного тестирования.

## 9. Процедуры контроля устойчивости

Для оценки устойчивости модель должна проверяться не только на “чистом” тесте, но и на стресс-сценариях.

### 9.1. Набор стресс-тестов

Рекомендуется проверять модель на:

- гауссовом шуме;
- blur;
- сниженной яркости;
- пересвете;
- JPEG-сжатии;
- частичной окклюзии;
- изменении масштаба объекта;
- поворотах и изменении ракурса;
- плотных сценах с перекрытием объектов;
- out-of-distribution изображениях.

### 9.2. Adversarial-aware тестирование

Если задача критична, дополнительно имеет смысл проводить:

- тесты с искусственными патчами;
- тесты на скрытые триггеры;
- тесты на чувствительность к небольшим возмущениям;
- проверку на резкие скачки confidence;
- сравнение поведения модели на естественных и искусственно искаженных примерах.

### 9.3. Регрессионное тестирование

После каждого изменения модели, датасета или инференс-пайплайна нужно запускать регрессию:

- на фиксированном benchmark-наборе;
- на наборе сложных кейсов;
- на наборе ранее обнаруженных ошибок;
- на robustness-поднаборах.

Если новая версия улучшает среднюю `mAP`, но резко ухудшает качество на критичном классе, такую модель нельзя считать безопасным улучшением.

## 11. Что считаем приемлемым качеством и устойчивостью

Конкретные пороги зависят от задачи и датасета, но логика принятия решения должна быть одинаковой.

Модель считается приемлемой только если одновременно выполняются условия:

- она показывает достаточное качество на чистом тестовом наборе;
- не проваливается на критичных классах;
- не имеет резкого падения на типовых стресс-сценариях;
- воспроизводимо работает в целевом окружении;
- не демонстрирует аномального поведения после экспорта и развертывания.

Иными словами, хорошая модель - это не просто модель с высокой `mAP`, а модель, которая:

- стабильно работает;
- адекватно переносится в продакшен;
- контролируется по данным и метрикам;
- проверена на типовые риски ML-систем.

## 12. Вывод

Модель `YOLOv8m` в задаче детекции объектов представляет собой не один файл с весами, а целую ML-систему, включающую данные, разметку, архитектуру, конфигурацию, инференс-пайплайн и процедуры оценки.

Ключевые активы системы - это датасет, аннотации, веса модели, конфигурация, preprocessing/postprocessing и инфраструктура исполнения. Именно эти элементы должны считаться критическими с точки зрения надежности и безопасности.

Основные риски для такой системы связаны с:

- ошибками и компрометацией данных;
- backdoor и poisoning-атаками;
- adversarial-примерами и патчами;
- подменой модели или конфигурации;
- сдвигом распределения данных после релиза;
- некорректной процедурой оценки качества.

Для управления этими рисками необходимо использовать не только стандартные метрики качества (`mAP`, `precision`, `recall`, `IoU`), но и процедуры контроля данных, robustness-тестирование, регрессионную проверку, мониторинг drift и эксплуатационные метрики.

Именно сочетание архитектурного понимания модели, контроля активов, анализа угроз и регулярной проверки метрик делает систему на базе `YOLOv8m` технически состоятельной и пригодной для практического использования.

## 1. Общая информация о файлах весов

В папке `weights` находятся два файла:

---

## 2. Что такое `.pt` файл в данном проекте

Файл `.pt` здесь представляет собой **PyTorch checkpoint**, созданный через механизм сериализации `torch.save(...)`.

Если говорить простыми словами, это контейнер, в котором сохранено состояние модели на определенный момент времени.

В вашем случае этот контейнер включает:

- структуру модели `DetectionModel`;
- веса всех слоев;
- EMA-версию весов;
- словарь с аргументами обучения;
- итоговые метрики;
- историю обучения по эпохам;
- служебные данные Ultralytics.

То есть файл можно рассматривать как "снимок" модели и части контекста ее обучения.

---

## 3. Физическая структура файла

Физически `best.pt` и `last.pt` являются ZIP-архивами формата PyTorch serialization.

Внутри архива находятся:

- `data.pkl` - сериализованная структура объектов Python;
- `byteorder` - информация о порядке байтов;
- `version` - версия внутреннего формата архива;
- множество файлов вида `data/0`, `data/1`, `data/2`, ... - бинарные блоки с tensor storage.

Для `best.pt` было извлечено:

- число записей в архиве: `483`
- число блоков tensor storage: `479`
- суммарный несжатый объем содержимого: `51969736` байт
- порядок байтов: `little`
- версия формата архива: `3`

Типичная структура выглядит так:

```text
best/data.pkl
best/byteorder
best/version
best/data/0
best/data/1
...
best/data/478
```

У `last.pt` структура такая же, только с префиксом `last/`.

Практический смысл этого следующий: PyTorch сохраняет не один текстовый конфиг, а набор сериализованных объектов и бинарных буферов, из которых потом восстанавливается модель.

## 8. Как устроена модель внутри checkpoint-а

Архитектура модели состоит из трех логических частей:

- `Backbone`
- `Neck`
- `Detect Head`

### 8.1. Backbone

`Backbone` отвечает за извлечение признаков из изображения.

В вашей модели backbone включает:

- последовательность `Conv`-слоев;
- блоки `C2f`;
- блок `SPPF`.

Смысл работы backbone:

- ранние слои выделяют простые признаки;
- средние слои собирают более сложные структуры;
- глубокие слои формируют семантически насыщенные признаки для детекции.

По встроенной конфигурации backbone выглядит так:

1. `Conv(64, 3x3, stride=2)`
2. `Conv(128, 3x3, stride=2)`
3. `C2f(128, repeats=3)`
4. `Conv(256, 3x3, stride=2)`
5. `C2f(256, repeats=6)`
6. `Conv(512, 3x3, stride=2)`
7. `C2f(512, repeats=6)`
8. `Conv(768, 3x3, stride=2)`
9. `C2f(768, repeats=3)`
10. `SPPF(768, kernel=5)`

### 8.2. Neck

`Neck` объединяет признаки разных масштабов.

Это нужно для того, чтобы модель могла лучше работать одновременно:

- с маленькими объектами;
- со средними объектами;
- с крупными объектами.

В вашей архитектуре `Neck` реализуется через:

- `Upsample`
- `Concat`
- дополнительные блоки `C2f`
- обратные downsample через `Conv(stride=2)`

То есть модель сначала поднимает признаки вверх по разрешению, объединяет их с более ранними слоями, а потом частично снова опускает.  
Это типичная многоуровневая схема fusion для YOLO.

### 8.3. Detect Head

Финальный модуль - это `Detect`.

Он получает признаки с трех уровней:

- из слоя `15`
- из слоя `18`
- из слоя `21`

Именно эти три уровня образуют три detection scale.

Параметры `Detect`:

- `nc = 5`
- `nl = 3`
- `reg_max = 16`
- `stride = [8.0, 16.0, 32.0]`
- `no = 69`

Поле `no = 69` означает число выходных значений на одну точку feature map.

Оно складывается так:

- `4 * reg_max = 64` значений на регрессию бокса;
- `5` значений на классы;
- всего `64 + 5 = 69`.

Внутри `Detect` есть две отдельные ветки на каждом масштабе:

- `cv2` - ветка регрессии bounding box;
- `cv3` - ветка классификации.

Для каждого масштаба структура веток выглядит так:

- `cv2[i] = [Conv, Conv, Conv2d]`
- `cv3[i] = [Conv, Conv, Conv2d]`

## 11. Чем этот файл отличается от обычного `state_dict`

Обычный `state_dict` - это в основном словарь имен параметров и тензоров.

Например:

```python

    'model.0.conv.weight': ...,
    'model.0.bn.weight': ...,
    ...
}
```

Но ваш `.pt` устроен значительно шире.

Он содержит:

- объект модели `DetectionModel`;
- встроенную конфигурацию архитектуры;
- EMA-веса;
- аргументы обучения;
- итоговые метрики;
- историю обучения;
- служебные метаданные Ultralytics.

Поэтому такие файлы правильнее называть не просто "весами", а **checkpoint-артефактами модели**.

Архитектурно это YOLOv8m с `depth_multiple = 0.67`, `width_multiple = 0.75`, `nc = 5`, тремя уровнями детекции (`stride 8/16/32`) и общим числом параметров `25,859,215`. Поэтому этот `.pt` файл следует рассматривать как полноценный checkpoint модели и эксперимента, а не как простой набор весов.
